# Taskplane Operator Console v1 Domain Model

## Status

Draft for TP-180.

## Purpose

Define the minimum shared domain model for the Operator Console so follow-on planning and UI work can distinguish:
- planning artifacts from execution artifacts,
- canonical files from derived views, and
- durable system-of-record state from convenience projections in the UI.

This model is intentionally conservative. It assumes Taskplane remains file-backed and that Operator Console views are projections over Taskplane-compatible files and artifacts rather than a replacement database.

## Modeling Principles

1. **Canonical state lives in files, not transient UI state.**
2. **Execution artifacts remain authoritative for runtime truth.**
3. **Planning artifacts may add context above execution, but must not duplicate runtime authority.**
4. **Derived UI views may combine multiple files, but should be reproducible from canonical sources.**
5. **Every important operator decision should be traceable to a file-backed artifact.**

## Core Entities

### 1. Idea
A lightweight statement of an opportunity, problem, request, or rough work candidate.

Typical fields:
- idea ID
- title
- summary/problem
- originating context/request
- status (proposed, exploring, accepted, rejected, parked)
- related project/initiative/spec links
- created/updated metadata

Role in the system:
- earliest planning-layer artifact
- may or may not turn into a formal spec or executable work

Expected storage posture:
- **Canonical file** in the future planning layer

### 2. Spec
A more concrete proposal describing intended behavior, scope, or design for a feature/change.

Typical fields:
- spec ID
- title
- objective/problem statement
- scope / non-goals
- acceptance criteria
- related ideas and initiatives
- related task packets
- status (draft, approved, superseded, delivered)

Role in the system:
- bridges planning intent to implementable execution work
- gives the operator and future tasks a stable reference

Expected storage posture:
- **Canonical file** in the future planning layer

### 3. Initiative
A grouping artifact that organizes multiple specs, workstreams, or milestones into a larger outcome.

Typical fields:
- initiative ID
- title
- outcome statement
- associated ideas/specs
- status (proposed, active, paused, complete)
- milestone/checkpoint summaries

Role in the system:
- provides project-level framing across multiple task packets and runs

Expected storage posture:
- **Canonical file** in the future planning layer

### 4. Task Packet
The executable unit of work already central to Taskplane.

Typical fields inferred from current task packets:
- task ID
- prompt/instructions
- status/checklist state
- dependencies
- execution notes/reviews
- area/location metadata
- completion markers and outputs

Role in the system:
- the handoff from planning context into execution
- the smallest durable work artifact the operator can launch/inspect

Expected storage posture:
- **Canonical file set** (packet folder and packet files such as `PROMPT.md`, `STATUS.md`, reviews, artifacts, and runtime-managed markers)

### 5. Batch
A coordinated execution grouping of task packets.

Typical fields inferred from prompts:
- batch ID
- included task packets
- wave/lane assignment
- lifecycle state
- timestamps and history
- integration/review results

Role in the system:
- primary runtime container for orchestrated execution across tasks

Expected storage posture:
- **Canonical execution artifact(s)** generated by Taskplane runtime

### 6. Run
A concrete execution attempt for a task packet or batch context.

Typical fields:
- run ID
- associated task packet and/or batch
- lane/worker identity
- start/end timestamps
- status/outcome
- logs and emitted artifacts

Role in the system:
- captures the actual execution event history
- supports operator diagnosis and auditability

Expected storage posture:
- **Canonical execution artifact(s)** generated by Taskplane runtime

### 7. Approval
A human decision point associated with a run, batch, task, or integration boundary.

Typical fields:
- approval ID
- subject (task, run, batch, integration request)
- requested decision
- decision state (pending, approved, rejected, expired, cancelled)
- approver identity or channel context
- evidence/artifact links
- timestamps

Role in the system:
- explicit record of human control in the loop
- enables Slack/web companion flows without moving authority out of Taskplane

Expected storage posture:
- **Canonical execution/supporting artifact** when backed by Taskplane/runtime files
- **Derived view** in the console when aggregated across sources

### 8. Artifact
Any durable file produced before, during, or after execution that matters to operators.

Examples:
- prompts and specs
- status files
- review files
- logs
- outputs/diffs
- integration evidence
- screenshots or reports if added later

Role in the system:
- evidence layer for traceability
- the bridge between high-level UI summaries and inspectable details

Expected storage posture:
- **Canonical file** when directly created/stored by the system
- **Derived attachment list/view** when grouped in the console

## Canonical Files vs Derived Views

## Canonical entities/files
These are the source-of-truth objects that should exist as files or runtime artifacts:
- ideas
- specs
- initiatives
- task packets (`PROMPT.md`, `STATUS.md`, reviews, packet-local artifacts)
- batch records/history
- run records/logs
- approval records/evidence when persisted by runtime flows
- produced artifacts

## Derived views
These are operator-console projections and must be recomputable from canonical files:
- backlog list/statuses
- task detail summaries that combine packet + run + approval info
- project dashboard rollups
- “ready / blocked / running / done” convenience statuses
- next recommended action hints
- approval inbox summaries
- recent projects or history views
- Slack notification payloads and web deep links

## Rule
If a view can be deleted and rebuilt from files without losing meaning, it is derived. If deleting it would destroy the official answer to “what happened?” or “what should execute?”, it is canonical.

## Relationship Model

### Planning-to-execution relationships
- **Idea -> Spec:** an idea may inspire zero, one, or many specs.
- **Spec -> Initiative:** a spec may belong to an initiative, though small specs may stand alone.
- **Spec -> Task Packet:** one spec may produce multiple executable task packets.
- **Initiative -> Task Packet:** an initiative may group many packets indirectly through specs or directly for reporting purposes.

### Execution relationships
- **Task Packet -> Batch:** a task packet may be included in zero or more batches over time, but each inclusion is a runtime event, not a redefinition of the packet.
- **Batch -> Run:** a batch may generate multiple runs/attempts or worker/lane execution records.
- **Task Packet -> Run:** a task packet may accumulate multiple run records across retries or different batches.
- **Run -> Approval:** a run may create zero or more approval requests.
- **Run/Batch/Task Packet -> Artifact:** each may produce logs, reviews, outputs, or integration artifacts.

### Cross-cutting relationships
- **Spec <-> Artifact:** specs may reference design documents and later delivery artifacts.
- **Approval -> Artifact:** approvals should link to the evidence used to make the decision.
- **Idea/Spec/Initiative -> History views:** planning artifacts may appear in console history, but history itself remains derived.

## Lifecycle Boundaries

### Planning layer boundary
Ideas, specs, and initiatives describe **intent**.
They answer:
- why should this work exist?
- how does this work fit into a larger effort?
- what outcome is desired?

### Execution layer boundary
Task packets, batches, runs, approvals, and runtime artifacts describe **operation and evidence**.
They answer:
- what work was actually packaged?
- what executed?
- what is running or blocked right now?
- what decision was requested or made?
- what artifacts were produced?

### Console layer boundary
Operator Console views describe **navigation and synthesis**.
They answer:
- what should the operator look at next?
- what items are ready, blocked, running, or waiting?
- how do planning and execution artifacts relate?

The console layer should never become the exclusive answer to those questions; it should expose the answer from the underlying files.

## Auditability and Source-of-Truth Rules

1. **Taskplane runtime artifacts are the source of truth for execution state.**
   - The console may summarize execution status, but batch/run/task execution truth comes from Taskplane-backed files and artifacts.

2. **Planning artifacts are the source of truth for planning intent.**
   - Ideas/specs/initiatives can add project meaning, but they do not redefine whether a task actually ran or completed.

3. **Derived statuses must document their inputs.**
   - For example, a backlog status such as “ready” or “blocked” should be derivable from packet metadata, dependency state, and runtime evidence—not manually edited in a separate store.

4. **Approvals must be traceable to evidence.**
   - An approval record should link to the task, run, batch, and relevant artifacts that justified the decision.

5. **UI-only edits are not canonical.**
   - The console may cache or render convenience state, but any meaningful mutation must round-trip back to a real file-backed or orchestrator-backed action.

6. **No duplicate authority.**
   - If the same field could be edited in both a planning artifact and a runtime artifact, define one as canonical and make the other a reference or projection.

7. **History should favor append-only evidence.**
   - Decisions, runs, and reviews should remain visible as a trace rather than being overwritten by the latest summary.

## Implications for Follow-on Tasks

This domain model implies:
- backlog and task detail views should be derived from packet + runtime state, not maintained separately;
- planning storage should introduce ideas/specs/initiatives as contextual files, not as a second runtime;
- Slack companion features should reference approvals and deep-link back to web views rather than own the workflow;
- project navigation should organize access to file-backed artifacts instead of inventing a hidden metadata store.

## Open Questions Deferred Beyond TP-180

- Exact file formats for approvals and project registry records
- Whether some runtime summaries should be materialized for performance while remaining reproducible
- How multi-project/workspace indexing should be cached without creating a second source of truth
