Runtime V2 is a redesign of Taskplane’s execution architecture that keeps the durable data plane intact while replacing the fragile runtime control plane. The specification says important file-backed contracts such as `PROMPT.md`, `STATUS.md`, `.DONE`, and `.pi/batch-state.json` remain authoritative, along with concepts like dependency waves, worktrees, and supervisor-driven orchestration. What changes is the runtime model around them: instead of relying on nested Pi sessions, TMUX infrastructure, and sidecar file polling, Runtime V2 moves to deterministic Node-owned processes with clearer ownership boundaries.

In the target architecture, the extension becomes a command surface, the engine owns orchestration decisions, each lane-runner owns serial task execution for one lane, and each agent-host owns exactly one Pi RPC subprocess. The architecture document emphasizes that every spawned process should have exactly one deterministic owner, which removes ambiguity about who is responsible for liveness, cleanup, and telemetry. Runtime V2 also formalizes mailbox-based steering, normalized event persistence, explicit packet-home path authority, and dashboard visibility that no longer depends on terminal pane capture.

Overall, Runtime V2 is meant to make Taskplane more deterministic, resumable, and observable without changing the core execution memory model users rely on. The design preserves checkpoint discipline, persistent worker context, review semantics, and orchestration state while eliminating correctness dependencies on TMUX and deprecated `/task`-only logic. Its success criteria focus on making `/orch` the authoritative execution path, ensuring dashboard and supervisor tooling work from structured runtime artifacts, and supporting stable single-task and multi-task orchestration through the same headless runtime core.
